## 互動 1：快問快答（5 min）

> **時機**：在 Demo 1（基礎指令）之後

### 題目 1（暖身題）
**Q：Docker Container 和虛擬機（VM）最大的差異是什麼？**

<details>
<summary>答案</summary>

Container 共享 Host 的 kernel，不需要安裝完整的 OS；VM 每台都有自己的 OS。
所以 Container 啟動只要幾秒，VM 要幾分鐘。

</details>

---

### 題目 2
**Q：我執行了 `docker run ubuntu`，但 Container 馬上就停了，為什麼？**

選項：
- A. Ubuntu image 壞掉了
- B. 沒有加 `-d` 參數
- C. Container 內沒有持續執行的程式
- D. 記憶體不夠

<details>
<summary>答案</summary>

**C**。Container 跑完預設指令就會結束。Ubuntu 的預設 CMD 是 `bash`，沒有 `-it` 的情況下 bash 馬上結束，Container 就停了。

要讓它保持運行：`docker run -it ubuntu bash`（互動模式）或 `docker run -d ubuntu sleep 3600`（背景模式）。

</details>

---

### 題目 3
**Q：`docker ps` 和 `docker ps -a` 有什麼不同？**

<details>
<summary>答案</summary>

- `docker ps`：只顯示**正在執行**的 Container
- `docker ps -a`：顯示**所有** Container，包含已停止的

這是很常見的新手困惑：「我的 Container 不見了？」→ 其實只是停了，用 `-a` 就看到了。

</details>

---

### 題目 4（加分題）
**Q：Docker Image 存放在哪裡？（兩個地方）**

<details>
<summary>答案</summary>

1. **本地**：你的電腦上（`docker images` 可以看到）
2. **遠端**：Docker Hub（或其他 Registry）

`docker pull` 從遠端拉到本地，`docker push` 從本地推到遠端。

</details>

## 互動 2：情境題挑戰（5 min）

> **時機**：在 Demo 3（Volume 持久化）之後

### 題目 1
**情境：你的同事把 MySQL Container 刪掉了，然後重新 `docker run` 一個新的。結果客戶的資料全部消失。請問問題出在哪裡？要怎麼預防？**

<details>
<summary>答案</summary>

**問題**：沒有使用 Volume。Container 內的資料在 Container 被刪除後就消失了。

**預防**：
```bash
# 使用 Named Volume 保存資料庫資料
docker run -d \
  -v mysql-data:/var/lib/mysql \
  mysql:8.0
```

只要 Volume 還在，就算 Container 被刪掉重建，資料也不會消失。

</details>

---

### 題目 2
**Q：以下兩個 Volume 用法有什麼差別？分別適合什麼場景？**

```bash
# 用法 A
docker run -v my-data:/app/data nginx

# 用法 B
docker run -v $(pwd)/src:/app/src nginx
```

<details>
<summary>答案</summary>

- **用法 A**：Named Volume（Docker 管理的儲存位置）
  - 適合：資料庫資料、需要持久化但不需要直接存取的資料

- **用法 B**：Bind Mount（指定 Host 上的路徑）
  - 適合：開發時同步程式碼，修改 Host 的檔案 → Container 內即時生效

記憶法：
- Named Volume = 把資料交給 Docker 保管
- Bind Mount = 把你的資料夾「共享」給 Container

</details>

### 題目 3
**Q：你跑了 `docker run -d -p 8080:80 nginx`，但瀏覽器打開 `localhost:8080` 看不到東西。你會怎麼排查？（說出至少 3 步）**

<details>
<summary>答案</summary>

排查步驟：

1. **`docker ps`** — Container 有在跑嗎？
2. **`docker logs <container>`** — 有沒有錯誤訊息？
3. **`docker port <container>`** — Port mapping 正確嗎？
4. **確認 8080 沒有被其他程式佔用** — `netstat -ano | findstr :8080`
5. **`docker exec -it <container> sh`** — 進去看服務是不是真的在跑

大部分問題都能在前 3 步找到原因！

</details>

## 互動 3：實戰判斷題（5 min）

> **時機**：在 Demo 4（Docker Compose）之後

### 題目 1（是非題）
**以下敘述對或錯？**

| # | 敘述 | 你的答案 |
|---|------|---------|
| 1 | `docker compose up -d` 會自動建立 network | ？ |
| 2 | `depends_on` 可以保證 MySQL 已經 ready 才啟動 app | ？ |
| 3 | `docker compose down` 會刪除 Volume 裡的資料 | ？ |
| 4 | 在 Compose 中，Container 之間可以用 service 名稱互連 | ？ |
| 5 | `.env` 檔案裡的變數只能用 `env_file` 才能讀取 | ？ |

<details>
<summary>答案</summary>

| # | 答案 | 說明 |
|---|------|------|
| 1 | ✅ 對 | Compose 自動建立一個 default network |
| 2 | ❌ 錯 | `depends_on` 只保證啟動順序，不保證 ready。要搭配 `healthcheck` + `condition: service_healthy` |
| 3 | ❌ 錯 | `docker compose down` 不會刪 Volume。要加 `-v` 才會 |
| 4 | ✅ 對 | 因為 Compose 自動建立 network 並提供 DNS 解析 |
| 5 | ❌ 錯 | `.env` 放在 `docker-compose.yml` 同層會自動被讀取，用於 `${VAR}` 變數替換 |

**重點提醒**：第 2 題是面試超級常考題！

</details>

### 題目 2（選擇題）
**你要部署一個 Node.js + PostgreSQL 的應用，以下哪種做法最好？**

- A. 把 Node.js 和 PostgreSQL 裝在同一個 Container 裡
- B. 分成兩個 Container，用 `docker run` 分別啟動，用預設 bridge network
- C. 分成兩個 Container，用 Docker Compose 管理，自訂 network + Volume
- D. 直接裝在 Host 上，不用 Docker

<details>
<summary>答案</summary>

**C**。

- A 錯：違反「一個 Container 一個職責」的原則，不好管理、不好擴展
- B 不好：預設 bridge 沒有 DNS 解析，管理多個指令也很麻煩
- **C 最佳**：Compose 統一管理，自訂 network 有 DNS，Volume 持久化資料
- D 不好：失去 Docker 的環境一致性和可移植性優勢

</details>

### 題目 3（開放題，如果時間充裕）
**回想一下你現在工作中的開發環境，有沒有哪些東西可以用 Docker 來改善？**

引導方向：
- 開發環境設定太複雜？→ Docker Compose 一鍵啟動
- 新人入職環境設定花太久？→ 把環境容器化
- 測試環境和正式環境不一致？→ 用同一個 Image 部署
- 需要同時跑不同版本的資料庫？→ 不同 Container 隔離

## 備用互動：如果時間多出來

### 現場挑戰賽

給學員 5 分鐘完成以下任務（如果學員有帶筆電）：

> **任務**：
> 1. 啟動一個 Nginx container，port 映射到 `3000:80`
> 2. 用 `docker exec` 修改 Nginx 首頁，把標題改成你的名字
> 3. 在瀏覽器確認修改成功
> 4. 停止並移除 Container

```bash
# 參考解答
docker run -d -p 3000:80 --name my-site nginx:alpine
docker exec -it my-site sh -c "echo '<h1>Hello, [Your Name]!</h1>' > /usr/share/nginx/html/index.html"
# 開 http://localhost:3000 確認
docker rm -f my-site
```

## 互動小技巧

1. **不要一次公布答案** — 先讓學員思考或討論 30 秒
2. **錯的答案也很好** — 「很多人都會這樣想，讓我解釋為什麼不是...」
3. **適時稱讚** — 「答對了！這題很多人會搞混」
4. **連結實務** — 每個答案都帶一句「所以在實際工作中...」
5. **觀察反應** — 如果大部分人答錯，代表這個概念需要再多講一點
